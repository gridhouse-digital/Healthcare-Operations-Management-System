# Make RLS isolation CI a real blocking gate

## Root Cause

The `rls-isolation` job was green only because the `Run two-tenant RLS isolation suite` step had `continue-on-error: true`.

The suite was not failing on any tenant-isolation assertion. Every test died during shared setup:

```text
createTenant(a): permission denied for table tenants
```

Local reproduction with Supabase CLI `2.106.0` showed the key capture was not the primary failure:

- `supabase status -o env` emits both legacy JWT keys and opaque keys.
- `SERVICE_ROLE_KEY` was already a JWT, so the existing opaque-key fallback was skipped.
- A REST insert into `public.tenants` with that key still returned `403`.
- Direct DB privilege check showed `service_role` authenticated correctly but had no table privileges:
  - `has_table_privilege('service_role','public.tenants','INSERT') = false`
  - `has_table_privilege('service_role','public.tenants','SELECT') = false`

The actual root cause is the latest Supabase local-stack default matching the new cloud behavior introduced on 2026-05-30: `api.auto_expose_new_tables` defaults to `false`, so a fresh `supabase db reset` no longer grants migration-created public tables to Data API roles. That left `service_role` unable to reach `public.tenants` through PostgREST, so the harness failed before RLS could be tested.

## Fix

- Set local `supabase/config.toml` `api.auto_expose_new_tables = true` so the fresh local CI database preserves the legacy Data API grants needed by the RLS suite.
- Keep the existing key-format diagnostics and opaque-key fallback, but update the misleading comment.
- Remove `continue-on-error: true` and the TODO comment from the RLS suite step, making the job blocking again.
- Document in `DECISIONS.md` that this is temporary: Supabase removes `auto_expose_new_tables` on 2026-10-30, and the durable fix is an explicit GRANT migration for intended API-role reachability.

RLS still enforces tenant isolation. The restored grants only make tables/views reachable through PostgREST; row visibility remains controlled by RLS policies and the suite's positive/negative tenant assertions.

## Evidence

Clean temp checkout from this branch, fresh local stack:

```text
npx supabase start
npx supabase db reset
deno task test:rls

ok | 68 passed | 0 failed
```

Before the config change on the same fresh local stack:

```text
FAILED | 1 passed | 67 failed
createTenant(a): permission denied for table tenants
```

## Notes

- This PR does not modify onboarding-gate feature code, migrations, `v_onboarding_gate`, or the recurring-compliance subsystem.
- A follow-up migration should replace the temporary local-stack compatibility flag with explicit GRANTs before 2026-10-30.
