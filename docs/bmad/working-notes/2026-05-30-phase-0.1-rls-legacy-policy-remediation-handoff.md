## Context
A live RLS gate run (the verification step recommended for the Phase 1 branch) surfaced a
**real cross-tenant leak** in `applicants` and `offers`. Architect verification confirmed it
and found the same pattern on objects the gate does **not** cover.

**Root cause:** PostgreSQL combines multiple *permissive* policies with **OR**. Epic 5 *added*
tenant-scoped policies but never *dropped* the Epic-0 allow-all policies, so
`OR(USING(true), tenant_match)` evaluates to `true` → zero isolation, regardless of the correct
policy sitting beside it.

**Important distinction:** the prior "Phase 0 gate MET" covered the **Edge-Function
`tenant_guard()`** layer (`docs\audits\phase-0-edge-function-tenant-guard-audit.md`). It did
**not** cover the **DB RLS-policy layer or storage**. This is that gap.

Verified live (still-existing) objects:

| Object | Stale policy | Gate-covered? | Sensitivity |
|---|---|---|---|
| `applicants` | `"Allow all access for authenticated users"` `USING(true)` (`20251128000000:20`) | Yes (flagged) | Candidate PII |
| `offers` | `"Allow full access for authenticated users"` `USING(true)` (`20251128000001:21`) **+** `"Everyone can view offers"` `auth.role()='authenticated'` (`20251129000002:63`) | Yes (flagged) | Offer / comp data |
| `ai_cache` | `"Authenticated users can read cache"` `TO authenticated USING(true)` (`20251209000000:35`) — not dropped by the Epic-5 ai_cache tenant migration | No | Cached AI output (may embed applicant data) |
| `ai_logs` | `"Authenticated users can read logs"` `TO authenticated USING(true)` (`20251209000000:16`) — legacy table, likely **no `tenant_id`** | No | AI prompt/response PII |
| storage: `resumes` + compliance-docs buckets | `"Authenticated users can view ..."` `TO authenticated`, no tenant scoping (`20251204000001`) | No (storage RLS not in the table suite) | **I9 / vaccination / license / background → PHI-class, highest severity** |

Confirmed **safe** (dropped in Epic 5, `20260309000001`): `employees`, `settings`,
`applicants_archive`, `offers_archive`, `profile_change_requests`.

**Owner decision (locked):** full-scope fix in a dedicated branch, landed first, with the
Phase 1 lifecycle branch rebased on top afterward.

## Scope
This brief **authorizes application/database code changes** (migrations + tests) for the
following only:
1. New migration that **drops** the stale permissive policies on live tables:
   - `applicants`: `"Allow all access for authenticated users"`
   - `offers`: `"Allow full access for authenticated users"`, `"Everyone can view offers"`
   - `ai_cache`: `"Authenticated users can read cache"`
   Use `drop policy if exists` (idempotent). Tenant-scoped policies already exist and remain.
2. `offers`: **keep** `"Allow public read access via secure_token"` (intentional candidate
   offer-link path) but **verify** it is token-scoped, not tenant-wide; tighten if it returns
   any offer regardless of token.
3. `ai_logs`: confirm whether still written (inspect `supabase\functions\_shared\aiClient.ts`
   and AI Edge Functions). If used → add `tenant_id` + tenant-scoped policy **or** restrict
   reads to `service_role`, and drop the authenticated-read policy. If dead → `drop table`
   with a documented rollback.
4. **Storage (top priority — PHI):** rewrite the `resumes` and compliance-docs bucket policies
   on `storage.objects` to enforce tenant scoping (tenant-id path prefix or metadata join);
   remove bare `TO authenticated` read access.
5. Extend the RLS test suite (`supabase\tests\rls\`) to cover `ai_cache`, `ai_logs` (if kept),
   and the storage buckets — closing the gate's blind spot.

**Deployment ownership (hard boundary):** this agent **stops at the green disposable-DB gate**
and hands back. It must **NOT** run `supabase db push` against the production/remote project
and must **NOT** merge to `main`. A separate agent (the one that originally surfaced the leak)

agent's deliverable is a **clean, deployable branch** with the migration written, the full RLS
suite green on a disposable DB, and the rollback block recorded in `DECISIONS.md` **before**

## Implementation Constraints
- **Multi-tenancy:** `tenant_id` only from `JWT → app_metadata → tenant_id`. Tenant-scoped
  policies must match the project pattern: `tenant_id = ((auth.jwt() -> 'app_metadata' ->> 'tenant_id')::uuid)`.
- **Root-cause discipline:** the fix is to **DROP** the legacy *permissive* policies. Do not
  "patch over" by adding more permissive policies. (If a policy must coexist with a broad rule,
  prefer `RESTRICTIVE` — but here dropping is the correct, simpler fix.)
- **Idempotency:** all `drop policy` / `create policy` statements use `if exists` / `if not exists`.
- **Audit:** any table that keeps/gains write access must retain its audit trigger; if `ai_logs`
  becomes tenant-scoped and writable by tenants, register it per `supabase\audit-tables.json`.
- Migrations only — no schema changes beyond what the scope lists; document rollback before `db push`.

## Security / Compliance Constraints
- This task **is** the tenant-isolation fix — treat cross-tenant read as a P0 defect on every
  listed object.
- **Storage / PHI:** the compliance-docs bucket holds I9 / vaccination / license / background
  documents. Cross-tenant exposure here is the highest-severity item. Confirm the corrected
  policy denies cross-tenant reads of stored objects, not just table rows. Flag for the owner
  whether storing these documents requires the BAA/HIPAA-PHIPA posture sign-off before this
  data is treated as production-grade.
- Never expose `*_encrypted` columns; never store signed URLs in the DB (regenerate from path).
- All writes to tenant-scoped tables remain audit-logged.
